03 专域 Agent 并行执行：子图检索 → 工具调用 → Policy 核验 → 置信度评估

06 任一 Agent 触发 escalate_flag=true → Orchestrator 统一升级人工

单 Agent 架构的三个瓶颈

串行处理延迟叠加
「掉线被判负 + 充值未到账」两个独立问题需串行处理，对局异常 Agent 与支付 Agent 无法并行，总时延 = 各子任务时延之和
专业深度与覆盖的矛盾
一个 Agent 同时处理退款规则、封禁申诉、英雄技能、版本强度，知识边界模糊，微调时各域样本相互干扰，精度难以持续提升
多语言规则膨胀失控
7 种语言 × 差异化 Policy × 不同支付渠道，单 Agent System Prompt 膨胀至 8000+ Token，推理成本激增且规则冲突难排查

三层 Agent 架构总览

L1 · 编排层（Orchestration Layer）
Orchestrator Agent
接收用户输入，完成 NLU 意图路由和任务拆解，判断单域还是多域并行，汇总子 Agent 结果，生成最终回复。不直接回答业务问题，只做调度。
Context Manager
跨 Agent 共享上下文：维护会话状态、全局槽位（uid/platform/language）、情绪分值。确保子 Agent 间信息一致，防止重复追问用户。
L2 · 专域执行层（Domain Agent Layer）— 5 个专域 Agent 并行
账号安全 Agent
处理：登录失败/账号封禁/被盗/申诉
图谱权限：Account/Punishment/Device
工具权限：账号系统 API（只读+申诉提交）
支付退款 Agent
处理：充值到账/退款/重复扣款
图谱权限：Order/Skin（is_used）
工具权限：支付系统 API（只读+退款提交）
对局补偿 Agent
处理：掉线判负/外挂/挂机举报
图谱权限：Match/Compensation/Punishment
工具权限：对局记录 API（只读）
游戏内容 Agent
处理：英雄技能/皮肤查询/赛季通行证
图谱权限：Hero/Skin/Season/Event
工具权限：公告 API（只读）

感知层 Perception

任务包解析
从 Orchestrator 接收 task_id / slots / context / timeout / lang 结构化输入，不自行解析原始用户文本
域内子意图精化
在 Orchestrator 路由基础上细化子意图（如退款→皮肤退款 / 充值退款），减少歧义
域内槽位补全
检测缺失必填槽位，向 Orchestrator 发起补充请求；不直接与用户交互，由 Orchestrator 统一追问
情绪状态接收
从 Orchestrator 接收情绪分值（0–1），自动调整回复语气阈值和工单优先级

推理层 Reasoning

知识图谱子图检索
只访问本域授权节点，Neo4j Cypher 多跳查询，最深 4 跳；跨域需求通过 Orchestrator 中转
Policy 规则获取
从规则政策 Agent 获取当前有效规则（含区服差异），不自行解释；接收推送更新后刷新本地缓存
ReAct 工具调度
最多 5 步工具链，每步检查是否已有充足证据，避免冗余 API 调用；超时 3s 熔断
置信度综合计算
综合图谱命中率、API 返回质量、规则匹配度，输出 0–1 置信分，<0.6 自动设置 escalate_flag

工具层 Tools

域专属 API（沙箱隔离）
每个专域 Agent 只能调用授权 API，运行时权限检查，越权调用直接返回 403，不降级处理
FAQ 向量检索（兜底）
置信度 <0.6 时触发，稠密检索 Top-3；游戏内容 Agent 有向量库写入权，其余只读
跨域数据请求
需要其他域数据时，向 Orchestrator 发出 cross_domain_request，不直接 Agent-to-Agent 调用，避免循环依赖
工具调用审计日志
每次调用记录 agent_id / tool_name / 参数摘要 / 耗时，写入不可篡改审计库，供合规审查

输出层 Output

结构化结果打包
返回 {task_id, answer_struct, confidence, evidence_ids, escalate_flag, action_items} 给 Orchestrator
不生成最终用户文本
输出结构化语义数据，由语言适配 Agent 负责生成面向用户的自然语言回复，保证本地化质量
操作指令打包
需执行的操作（退款提交、补偿发放）打包为 action_items，等 Orchestrator 确认后统一执行，防止重复操作
升级标记与原因
escalate_flag=true 时附带 escalate_reason（置信度不足 / 权限不足 / 情绪超阈值 / 命中升级规则）

多轮对话状态机

运行于 Orchestrator Agent · 8 个对话状态 · 跨专域 Agent 共享上下文 · 情绪与语言自适应

IDLE · 空闲等待
初始状态，等待用户发起对话，无任何上下文槽位
用户发消息 → INTENT_DETECTION
INTENT_DETECTION · 意图识别
NLU 解析用户输入，提取意图类型和已有实体槽位
置信度 >0.85 且槽位完整 → RETRIEVAL
置信度 >0.85 但槽位缺失 → SLOT_FILLING
置信度 0.6–0.85 → CLARIFICATION
置信度 <0.6 → FAQ_FALLBACK
检测到高情绪 → EMOTION_HANDLING
SLOT_FILLING · 槽位追问
意图明确但关键槽位缺失，主动追问用户补充信息。每次只追问一个最关键槽位，最多追问 3 轮
用户补充了目标槽位 → RETRIEVAL
用户换了话题 → INTENT_DETECTION
追问超过 3 轮仍缺失 → HUMAN_ESCALATION
用户明确拒绝提供 → FAQ_FALLBACK
RETRIEVAL · 多 Agent 检索
槽位就绪，Orchestrator 拆解任务后并行派发至专域 Agent，等待所有子任务返回后合并结果
单域任务，专域 Agent 返回高置信结果 → RESPONSE_GENERATION
多域任务，所有 Agent 均返回 → RESPONSE_GENERATION（合并结果）
某专域 Agent 需补充槽位 → SLOT_FILLING
专域 Agent 超时 / 工具调用失败 → FALLBACK（降级处理）
命中升级规则（任一 Agent 判断） → HUMAN_ESCALATION
CLARIFICATION · 意图澄清
置信度中等，Agent 给出 2–3 个可能意图供用户确认，避免答非所问
用户选择了某个意图 → SLOT_FILLING / RETRIEVAL
用户重新描述问题 → INTENT_DETECTION
用户仍不明确 → FAQ_FALLBACK
RESPONSE_GENERATION · 生成回复
基于检索结果选择 LLM 生成或模板填充，附加操作引导和满意度询问
回复发送后用户满意 → IDLE（会话关闭）
用户追问或问题未解决 → INTENT_DETECTION
用户要求人工 → HUMAN_ESCALATION
EMOTION_HANDLING · 情绪安抚
检测到用户高情绪（愤怒/焦虑）时进入，优先安抚再解决问题，暂停所有技术性追问
情绪缓解（1–2轮交互后） → INTENT_DETECTION
用户持续激动超 3 轮 → HUMAN_ESCALATION（P1）
用户明确要求人工 → HUMAN_ESCALATION
HUMAN_ESCALATION · 升级人工
生成工单预填内容，标注优先级，路由到对应技能组，会话进入人工接管状态
人工接入成功 → 终态（人工接管）
排队超时无人工 → 留存工单 + 异步回调

数据飞轮设计

人工处理 → 数据沉淀 → 知识更新 → Agent 优化 → 减少人工 · 形成正向循环

💬
1 · 用户发起咨询
用户通过 App 内客服、微信公众号、官方论坛等渠道发起咨询，Agent 记录完整对话流（意图、槽位、上下文、情绪分值）和处理结果（自动解决 / 升人工 / 用户放弃）。
多渠道接入全量日志匿名脱敏
🤝
2 · 人工处理与标注
Agent 无法自动解决的工单流转至人工。人工客服在处理时同步完成数据标注：确认真实意图类别、补充槽位值、记录正确答案和解决方案。质检人员复核标注质量（抽样率 20%）。
意图修正槽位补全答案记录质检抽样
🔍
3 · 自动挖掘高价值样本
每日凌晨数据管道自动运行：筛选 Agent 处理失败（置信度 <0.6）但人工成功解决的工单，这些是最有训练价值的"难例"。同时识别新出现的问题类型（无对应意图的工单聚类）作为扩充候选。
难例挖掘新意图发现聚类分析日均处理
📚
4 · 知识图谱与 FAQ 更新
高质量样本经审核后分两路更新：（1）结构化问题（有明确节点归属）更新到知识图谱对应节点的属性或新增 Issue 类型节点；（2）长尾问题（无图谱对应）写入 FAQ 向量库并向量化索引。Policy 变更走独立审批流程。
图谱节点更新FAQ 向量入库双路分发
🧠
5 · 模型增量微调
当某意图累计新样本 >200 条，或准确率连续3天低于基线时，触发 NLU 模型增量微调。微调采用 LoRA 轻量级方式，在原模型基础上叠加，不影响已有意图的性能。微调后经 A/B 测试验证效果。
LoRA 微调A/B 验证增量不破坏存量
🚀
6 · 灰度部署与效果验证
更新后的知识和模型走灰度发布流程（5% → 30% → 100%），实时监控自动解决率、满意度、升人工率三大核心指标。效果提升则全量，效果回退则触发熔断回滚。
灰度发布指标监控自动熔断
📈
7 · 自动解决率提升 · 飞轮加速
Agent 能力增强后，能自动解决更多问题，人工处理量减少。但人工处理的都是更难、更边缘的案例，标注价值更高，进一步驱动下一轮优化。飞轮越转越快，理论天花板是 95% 自动解决率（保留 5% 给高情绪和复杂纠纷）。
目标 95% 自动化边际成本递减持续自我进化

交互效应

节省 20%

输出长度控制
节省 15%
Prompt 中明确要求「回复不超过 150 字」，避免模型冗余输出。操作步骤类上限 200 字，FAQ 类上限 100 字

批处理与调度优化

低峰批处理任务
节省 30%
FAQ 向量化、知识库更新、日报生成、补偿批量发放等非实时任务，统一调度到凌晨 01:00–06:00 执行，利用 API 低峰定价
语义缓存（Semantic Cache）
节省 18%
相似度 >0.95 的问题复用上一次 LLM 回复（GPTCache），高频重复问题（「赛季什么时候结束」）命中率高达 60%
Embedding 缓存
节省 12%
FAQ 向量化结果缓存至 Redis，知识库更新前不重复计算。只有新增/修改条目才触发重新向量化
流式输出优化
降低延迟
启用 Streaming 模式，用户感知首 Token 时延 <400ms，无需等待完整回复，提升主观体验
